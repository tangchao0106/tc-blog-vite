解码策略：如何"选词"决定输出质量 ​

大模型吐出一个字还是吐出一句话，背后是一套关于"怎么选"的算法博弈。

开头：一个字怎么蹦出来的 ​

你有没有想过，当你对大模型说"写一首诗"，它是怎么一个字一个字往外"吐"的？

说白了就是循环预测：每生成一个新词，就把前面生成的所有词当成"提示"，让模型重新算一遍——下一个词最可能是什么。这个过程叫做自回归生成（Autoregressive Generation），就像你玩猜词游戏，每次给一个提示，你猜一个词，下一轮再根据这个词继续猜。

但问题来了：每次猜词的时候，候选词可能有很多个，模型给每个候选词都打了个分（概率），怎么从这些候选词里挑一个？

这就是解码策略要解决的问题。不同的选法，出来的文字质量天差地别。

贪婪解码：永远选"最优"的那个 ​

最直观的做法：每次选概率最高的那个词。这叫贪婪解码（Greedy Decoding）。

想象你站在一个分岔路口，每条路都标着"这条路通向正确答案的概率"。贪婪的人就是：哪条路概率最高就走哪条，不回头。

text
模型说："床前明" → "月"（概率最高 0.95）→ "光"（0.92）→ "疑"（0.88）→ ...

优点：

  • 简单高效，每次只算一次选择
  • 结果确定性高，不会随机跳脱

致命缺点： 这就像一个从来不走回头路的人，一旦选错了一条岔道，就再也到不了真正的目的地。在文字生成里，这意味着——

贪婪解码容易陷入局部最优陷阱。它会反复生成"安全的废话"，比如 ChatGPT 早期的"综上所述""首先其次最后"——因为这些词的概率确实高，但满屏都是就无聊了。

什么时候用它：对速度要求极高、质量要求一般的场景（比如填空、检索排序）。

束搜索：同时走N条路 ​

束搜索（Beam Search）的思路是：别只留一条路，留N条"最有可能的路"一起走。

你可以理解为：你派了N个探险队同时出发，每到分岔路口，每个队伍都只保留自己这条路上概率最高的分支。这样既能探索更多可能性，又不会像完全随机采样那样跑偏。

text
Beam Size = 3（同时走3条路）
第1步：["床"(0.95), "桌"(0.03), "楼"(0.02)] → 保留前3
第2步：分别扩展，找到每条路的最佳下一词 → 保留综合评分前3的路径
...

为什么叫"束（Beam）"：可以想象成一个扇形的搜索窗口，像手电筒的束光一样，照亮的范围有限，但比一根针宽。

实际效果：

  • 生成质量通常比贪婪好很多
  • 是大多数翻译、摘要系统的默认选择
  • 平衡了质量和效率

注意：Beam Size 越大，结果越好吗？不一定。太大的束宽会让生成变得保守——因为所有分支都在追求"安全的高概率词"，反而丢失了创造性。

采样：从"确定性"到"掷骰子" ​

到这里你可能发现了：贪婪和束搜索都是确定性的——同样的输入，永远得到同样的输出（或者只有微小变体）。这在很多场景下是好的，但用来对话、写故事、生成代码就太无聊了。

于是引入了采样（Sampling）：按概率大小来"掷骰子"选词。

为什么按概率？ ​

简单理解：模型给"月"打了95%的分，给"桌"打了3%，给"楼"打了2%——那就让"月"有95%的概率被选到，而不是绝对地选它。掷骰子的结果虽然有随机性，但长期来看概率分布是不变的。

这样文字就有了变化和创意：同一句诗的提示，今天可能生成"床前明月光"，明天可能生成"床前明月光"，但偶尔也会生成不那么"正确"但更有创意的词。

Top-k 采样：只允许骰子落在"前K名" ​

纯采样有个问题：如果概率分布很平（比如每个候选词概率都是1%、2%），骰子可能会掷出一个完全离谱的词——虽然概率低，但终究可能被选中。

Top-k 采样就是在掷骰子之前先做一次筛选：只保留概率最高的前K个词，其余的直接忽略，然后在这K个词里按概率采样。

text
候选词概率：[月:0.35, 光:0.25, 霜:0.15, 雪:0.10, 星:0.08, ...]
Top-k = 3 → 只看 [月, 光, 霜]，再按概率分配骰子
→ 月的概率 = 0.35/(0.35+0.25+0.15) ≈ 47%

k 设成多少合适：

  • k 太小（比如 k=1）→ 等同于贪婪，容易重复
  • k 太大（比如 k=10000）→ 接近纯采样，可能跳词
  • 实际经验：k=20~100 是常见区间

适合场景：需要多样性的创意写作、头脑风暴式对话。

Top-p（核采样）：更动态的"骰子范围" ​

Top-k 有一个硬伤：不管候选词分布是陡峭还是平坦，K都是固定的。

想象两种情况：

  1. 模型非常有把握：概率集中在前3个词 [0.95, 0.03, 0.01]
  2. 模型很纠结：概率平摊在前100个词 [0.02, 0.019, 0.018, ...]

用同样的 k=50，第一种情况下会混入大量低概率词，第二种情况下会筛掉很多其实也合理的词。

Top-p（也叫 Nucleus Sampling） 就是来解决这个问题的：

不是固定选前K个，而是从概率最高的词开始往下累加，累加到概率之和超过阈值 p（比如0.9），就停。这个动态的候选集合叫"核（nucleus）"。

text
p = 0.9
候选：[月:0.35, 光:0.25, 霜:0.15, 雪:0.10, 星:0.08, 风:0.04, ...]
累加：0.35 → 0.60 → 0.75 → 0.85 → 0.93 → 超过0.9！
→ 核集合 = [月, 光, 霜, 雪]（这4个词的累积概率刚好超过90%）
→ 只在这4个词里按概率采样

为什么更好：

  • 模型自信时，核很小，输出稳定
  • 模型纠结时，核变大，允许更多变化
  • 自适应地平衡"质量"和"多样性"

这是目前最主流的采样策略，大多数产品级对话都默认用 Top-p。

温度：控制骰子的"偏心程度" ​

最后来说一个控制"随机性"的全局旋钮——温度（Temperature）。

温度这个概念来自统计力学。放到语言模型里，它的作用是调整概率分布的"尖锐程度"：

  • T = 1.0：概率分布不变，完全按模型原意采样
  • T < 1.0（比如0.7）：概率分布变陡峭——高分词概率更高，低分词概率被压低。输出更确定、更保守、更像"正确答案"
  • T > 1.0（比如1.5）：概率分布变平坦——各个词的概率差距缩小，骰子更容易掷到"意外"的词。输出更有创意，但也可能更离谱
text
原始概率：[月:0.95, 光:0.03, 雪:0.01]
经过 softmax 温度调整后：
T=0.5 → [月:0.999, 光:0.001, 雪:0.0001] → 几乎只会选月
T=1.5 → [月:0.85, 光:0.10, 雪:0.04] → 偶尔能跳出月

实际应用建议：

  • 写代码、做推理、事实问答 → 低温度（0.3~0.7）
  • 聊天、头脑风暴、创意写作 → 中温度（0.7~1.0）
  • 要求极度多样性（比如生成多种备选方案） → 高温度（1.0~1.5）

大多数 API（比如 OpenAI）把温度参数暴露给用户，让你自己调旋钮。

解码策略组合使用 ​

实际产品里，这些策略往往是组合使用的：

Top-p = 0.9, Temperature = 0.8
= 先用温度调整概率分布，再用 Top-p 动态选候选池，最后在池内采样

这种组合是目前 GPT-4、Claude 等主流模型对话时的默认配置。

还有两个值得了解的补充策略：

  • 重复惩罚（Repetition Penalty）：如果某个词被重复使用，降低它后续被选中的概率，解决"大模型复读机"问题
  • 长度惩罚（Length Penalty）：防止束搜索偏向生成过短或过长的句子

总结：一场"质量 vs 多样性"的博弈 ​

解码策略的核心矛盾其实很简单：

策略质量倾向多样性速度
贪婪稳定但保守低最快
束搜索较好中快
Top-k 采样依赖k设置可调快
Top-p + 温度最好（通常）最灵活快

没有绝对的"最好"，只有"最适合当前场景"。写代码追求稳定准确，头脑风暴追求出其不意——你拧的旋钮，决定了模型是"一本正经的专家"还是"天马行空的诗人"。

理解这些解码策略，下次你调 API 的 temperature 参数，或者看到模型输出"太保守/太跳脱"时，就能知道问题出在哪里了。
