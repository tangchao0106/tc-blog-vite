RAG：让大模型"查资料"再回答 ​

大模型的"知识盲区" ​

你有没有遇到过这种情况：问大模型一个很专业的问题，它一本正经地给你编了一个答案？比如问它"我们公司去年 Q3 的净利润是多少"，它会说"根据公开数据，约为 3.2 亿元"，说得跟真的一样，但完全是在胡编。

这不是大模型不努力，而是它的"知识"有截止日期——它只会回答训练数据里有的东西，而且对私有知识（公司内部文档、私人笔记、非公开数据）天生就不知道。

那怎么办？

一个最直觉的思路是：不知道就查嘛。人类遇到不会的问题，第一反应也是去查资料。大模型也应该有这个能力。

这就是 RAG（Retrieval-Augmented Generation，检索增强生成） 要解决的问题——给大模型接上一个"外部知识库"，让它在回答之前先查一查资料，然后基于查到的内容组织答案。

RAG 三步走：找料 → 筛选 → 回答 ​

RAG 的流程用一句话概括就是：检索 → 精排 → 生成。拆开来看，分成三个环节：

第一步：检索（Retrieval）—— 大海里捞针 ​

检索的任务是：从一堆文档里，快速找到和用户问题最相关的几个片段。

这一步的核心是把文本变成数字。因为计算机处理"文字"不如处理"数字"得心应手，所以要把文字映射到一个高维向量空间——这个过程叫Embedding（嵌入）。

你可以理解为：Embedding 就是把每段文字翻译成一段坐标。比如"苹果手机很好用"和"iPhone 用起来很流畅"这两句话意思相近，它们翻译出来的坐标距离就很近；而"今天天气不错"和它们的距离就非常远。

向量数据库（Milvus、Chroma、Qdrant 等）就是专门干这个的——存向量 + 做相似度搜索。给用户问题也生成一个向量，然后在数据库里找和它距离最近的 Top-K 个文档片段。

第二步：重排序（Rerank）—— 让最好的排在前面 ​

问题来了：向量检索找出来的结果，真的就是最相关的吗？

不一定。Embedding 模型本质上是在做"语义近似"，但它有时候会被表面相似性欺骗。比如用户问"特斯拉 2023 年的营收"，Embedding 可能找到的是"特斯拉 2022 年的财报摘要"，语义上非常接近，但时间不对。

重排序就是来解决这个问题的。

重排序模型（Cross-Encoder 类）会更仔细地比较用户问题和每个候选文档的相关性。它不是把两者都压缩成向量来比距离，而是把"用户问题 + 文档"作为一对输入，完整地喂进模型，让模型直接输出一个相关度分数。

你可以把向量检索理解成海选，用速度快但精度一般的Embedding做快速过滤；重排序理解成复赛，对前几名做精细化评估，重新排个序。

第三步：生成（Generation）—— 带着答案写文章 ​

有了精筛后的参考资料，大模型就可以正式上场了。

它把用户的问题 + 检索到的文档片段一起打包成 Prompt，格式通常长这样：

你是一个专业助手。请根据以下参考资料回答用户问题。如果资料中没有相关信息，请如实说明。
    
参考资料：
1. [文档片段1...]
2. [文档片段2...]

用户问题：xxxxx

然后大模型基于这些"临时喂进去的知识"来组织答案，而不是靠自己的"记忆"（参数权重）。

为什么要费这么大劲？直接微调不行吗？ ​

你可能有个疑问：既然模型不知道这些知识，那我直接拿这些文档去微调（Fine-tuning）不就好了？

这个问题问得好。实际上，微调和 RAG 是两条不同的路，各有适用场景：

微调适合：你希望模型永久学会某种能力（比如特定的写作风格、推理模式、专业领域的思维方式）。微调后模型"内化"了这些知识，响应快，不需要额外检索。但微调成本高，而且更新知识需要重新训练。

RAG 适合：你需要模型随时访问最新或海量的私有知识。知识是动态变化的（每天都有新文档），或者知识库太大（几亿条记录）塞不进模型里。RAG 的优势是知识独立存储，改知识不用改模型。

打个比方：微调像是把知识刻进脑子里，RAG 像是把知识存在图书馆里——需要的时候去查。

现实中最常见的做法是RAG + 微调组合使用：用微调让模型学会怎么"正确使用"检索结果，用 RAG 给它提供实时、准确的知识。两条腿走路。

RAG 的几个工程难点 ​

光知道流程还不够，真正落地时会遇到几个典型坑：

Embedding 模型选错了 ​

Embedding 模型直接决定检索质量。如果模型本身对专业术语理解差，检索出来的结果就牛头不对马嘴。

选型建议：中文场景下 MiniLM、Text2Vec、m3e 等开源模型效果不错；追求更高精度可以用 OpenAI 的 text-embedding-3-large 或者 Cohere 的 Embed 系列。关键是要选在和你的业务领域相近的数据上训练或微调过的模型。

文档切分是个技术活 ​

一个 PDF 可能有 100 页，是切成 100 份还是 10 份？每份多少 token？

切得太碎：每个片段上下文太小，模型看不到足够的信息来回答问题。

切得太粗：包含了大量无关信息，把真正相关的内容给稀释了，而且容易超过模型的上下文窗口。

常见的策略有：

  • 固定窗口切分：每 N 个 token 一切，简单粗暴，但容易在句子中间断开
  • 语义切分：按段落、标题结构来切，保留语义完整性
  • 递归切分：先按大段落切，如果太长再递归细分
  • Agentic RAG：让模型自己决定在哪里切，完全动态（前沿方案，但成本高）

混合检索：别只靠向量 ​

纯向量检索有个问题：它擅长"语义相似"，但不擅长"精确匹配"。

比如用户搜"订单号 ORD-20240615-001"，精确匹配才有用，向量检索大概率找不到。

解决方案是混合检索：向量检索 + 关键词检索（BM25）并行跑，然后用一个 Reciprocal Rank Fusion（RRF）算法把两个结果合并。两种检索方式互补，各自的弱点被对方的优势补上了。

检索结果噪声 ​

就算检索到相关文档，里面也可能混入了干扰信息。大模型的 Attention 会平均分配给所有输入 token，无关内容太多会"冲淡"真正重要的信息。

应对策略：给检索到的内容加元信息标记（来源、时间、相关性分数），让模型知道哪些更可信；也可以在 Prompt 里明确要求"只根据以下材料回答，不要自行发挥"；更进阶的做法是用 Attention 机制过滤噪声，或者在精排阶段就做质量筛选。

RAG 现在进化到哪了？ ​

RAG 不是一个静态的技术，它在快速演进。几个值得关注的趋势：

Self-RAG（斯坦福）：让模型自己判断是否需要检索，以及检索到的内容是否有用。模型在生成过程中会插入一些特殊的反思 token 来自我评估，这比盲目检索要聪明得多。

Corrective RAG：检索结果不是直接用，而是先让模型评估"这个资料靠谱吗"，不靠谱就重新检索或者改用搜索引擎。这有点像是给 RAG 加了一个"质量审核员"。

Agentic RAG：不再是简单的"查一次"流程，而是让 Agent 循环执行"检索 → 判断 → 再检索 → 再判断"，直到得到满意的结果。这对付复杂的多跳问题特别有效。

知识图谱 + RAG：把文档里的实体和关系抽取出来构建知识图谱，检索时不仅找相关段落，还沿着知识图谱的边做关联推理。比如问"谁是谁的老板"，知识图谱能顺着关系链找到答案，比纯向量检索更精准。

什么时候该用 RAG？ ​

最后给个简单的决策框架，帮你判断是否需要 RAG：

场景适合用 RAG适合用微调
知识动态变化（每天更新）✅ 必用❌ 不适合
海量私有文档（10万+条）✅ 天然适配❌ 放不下
需要实时数据（股价、库存）✅ 实时查询❌ 做不到
固定领域的专业术语和逻辑二选一或组合二选一或组合
特定写作风格或语气❌ RAG 帮不上✅ 微调更稳

写在最后 ​

RAG 的本质是什么？它是把知识的存储和知识的运用分开。

模型负责"思考"和"表达"，知识库负责"记忆"和"查询"。各司其职，动态组合。这个解耦让大模型从"通用知识库"变成了"可插拔的专业助手"。

搞清楚 RAG 的三步流程（检索→精排→生成），理解向量检索和关键词检索的区别，知道切分和噪声这两个坑——你就掌握了 RAG 的核心。至于 Self-RAG、Agentic RAG 这些进阶玩法，等你基础打牢了再去探索也不迟。
