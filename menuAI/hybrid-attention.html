混合注意力：滑动窗口与全局注意力的"精读与概览" ​

你读这篇文章的时候，其实在做两件事：精读当前这句话，同时扫一眼前后段落知道上下文在讲什么。这两件事的认知负担完全不同——精读需要逐字分析，扫读只需要知道大概在说什么。

大模型的注意力机制也面临同样的问题。如果把所有 token 一次性做全局注意力（每两个词都要互相"打招呼"），计算量会爆炸；但如果只盯着最近几个词看，又容易丢了远距离的上下文关系。

所以聪明的研究者想出了一个办法：混合注意力——让模型同时拥有"精读"和"概览"两种模式。这篇文章就来讲讲这个思路。

想象一个双轨阅读器 ​

你可以把混合注意力理解为一个双轨阅读器：

  • 滑动窗口注意力（SWA）：每次只看固定的窗口范围，比如前后各 512 个 token。好比读一本书时，眼睛每次只聚焦在当前段落附近，远处的章节暂时不看。
  • 全局注意力（GA）：偶尔停下来，翻回去看目录、前言，或者联系开头埋的伏笔。这是全局的、跨越整篇文章的关联。

两种注意力并行工作，各司其职。局部细节由滑动窗口处理，长距离依赖由全局注意力搭桥。

滑动窗口注意力：看得近，但看得清 ​

标准的全注意力（Full Attention）会把序列中任意两个 token 都计算一次关联度。如果序列长度是 n，那attention 的计算复杂度是 O(n²)——长度翻倍，计算量翻四倍。这在大模型里是个灾难。

滑动窗口注意力把每个 token 的视野限制在固定窗口大小 w 内。比如 GPT-4 的某个变体用 8192 的窗口，你当前位置的 token 只能和前后各 8192 个 token 交互，远处的 token 暂时被"忽略"。

打个比方：你在一个图书馆里查资料，全注意力相当于每次借阅都要向全馆广播"我想找所有书里和这本最相关的内容"；滑动窗口相当于你只向你左右相邻的几排书架去问，效率高多了。

计算复杂度从 O(n²) 降到了 O(n·w)，而 w 通常远小于 n。

但这里有个问题：信息怎么从第 1 个 token 传到第 10000 个 token？ 如果只有滑动窗口，远处的 token 永远无法影响当前 token——模型就成了"近视眼"。

全局注意力：偶尔拉通全场的"大会" ​

全局注意力的出现就是为了解决长距离依赖。它允许某些特殊的 token（通常是特殊标记，如 [CLS]、[G] 或者第一层的某些 token）能够看到序列中的所有其他 token，并被所有 token 看到。

你可以理解为：模型里安插了几个"记者"token，它们分布在序列的关键位置，任务就是把所有信息收集起来，然后广播给其他人。这些记者 token 就像神经网络里的 hub 节点——它们的注意力范围是全局的。

实际上，许多现代大模型（如 Mistral、Qwen 等）会在每一层都使用滑动窗口注意力，同时在某些特定层或特定 token 上加入全局注意力能力。每一层的 SWA 负责局部精读，逐层往上传递信息；全局注意力负责远距离的"大局观"。

两种注意力的融合方式 ​

混合注意力不是简单地把两种注意力拼在一起，而是有几种典型的融合策略：

1. 交替堆叠（Stacking） Transformer 的某些变体（如 Longformer）会让不同的注意力层交替使用 SWA 和 GA。低层用滑动窗口提取局部特征，高层用全局注意力整合远距离信息。就像人读书，先逐段细读，最后整体回顾总结。

2. 膨胀滑动窗口（Dilated Sliding Window） 受膨胀卷积（Dilated Convolution）的启发，跳过某些 token 形成"跳跃式"窗口。比如只看第 1、5、9、13... 个 token，虽然窗口内 token 少了，但覆盖的感受野反而更大了。这比固定窗口更高效地捕捉不同范围的依赖关系。

3. 全局 token + 局部窗口的组合 比如 BigBird、Longformer 这类模型，在序列里插入少量全局 token（如 [CLS]），它们能直接看到所有 token；同时普通 token 只在局部窗口内交互。信息通过全局 token 像"中转站"一样流通。

为什么要这样做？聊聊计算量的账 ​

我们来算一笔账，直观感受下混合注意力的价值：

假设序列长度 n = 16384，窗口大小 w = 512：

  • 全注意力：16384 × 16384 ≈ 2.68 亿 次计算
  • 滑动窗口注意力：16384 × 512 ≈ 840 万 次计算

差了约 32 倍！而效果上，对于大多数自然语言任务，相邻 token 之间的关联本来就占大多数，全窗口的全局注意力有点"杀鸡用牛刀"的感觉。

不过，如果任务真的需要跨越超长距离的逻辑关联（比如对比文章开头和结尾的对应关系），纯滑动窗口就会捉襟见肘。这也是为什么需要全局注意力来兜底。

实际应用：你的模型已经在用它了 ​

你可能已经在用搭载混合注意力的模型了，只是不知道：

  • Mistral 7B 用的 Grouped-Query Attention（GQA）配合滑动窗口，能在 4096 上下文上高效运行。
  • Qwen2 系列在某些配置中使用了局部注意力 + 全局注意力的混合结构。
  • Gemini 的架构据传也大量使用稀疏注意力来处理百万级别的上下文。

现代长上下文模型（如支持 100K、1M token 上下文的模型），几乎都用到了混合注意力或类似的稀疏注意力变体。纯全注意力在那个长度上根本跑不动。

怎么理解这种设计的直觉 ​

把混合注意力想成一个高效团队：

  • SWA 是一线员工，负责处理手头具体的任务，信息在团队内部高效流转。
  • GA 是项目经理，偶尔召集全员大会，把散布在各处的关键信息汇总起来。

两者配合，才做到了既高效又有大局观。这其实是工程上很典型的"分而治之"思路：大事化小，小事化了，局部搞定了再全局整合。

总结 ​

混合注意力的核心洞察是：不是所有 token 之间的关系都同等重要。相邻词之间要精读，远距离词之间偶尔拉通就够了。通过滑动窗口 + 全局注意力的组合，模型在计算量和建模能力之间找到了一个很好的平衡点。

理解了这个设计，你再看到"这个模型支持 128K 上下文"之类的宣传时，就能知道背后大概率不是傻傻的全注意力——而是巧妙的稀疏注意力在撑着。
