KV Cache优化：从PagedAttention到MLA ​

开篇：显存去哪了？ ​

你有没有这种感觉：大模型生成回答时，越往后越慢？开头几秒一个字一个字蹦，后来干脆卡住不动了。这不是模型"累了"，而是显存不够用了。

说白了，大模型推理时最吃显存的不是模型参数（那些权重在推理前就加载好了），而是 KV Cache——每个生成步骤里用来"记忆"之前所有词的中间计算结果。

今天我们就来拆解这个问题：KV Cache 是什么，为什么它成了瓶颈，以及 vLLM、DeepSeek 等主流框架是怎么"省内存"的。

KV Cache：推理时的"草稿纸" ​

先用一个生活场景让你理解 KV Cache 的本质。

想象你写一篇长文章，不是从头一口气写完，而是每写一个字，都要回头把前面所有字重新读一遍，然后才能决定下一个字该写什么。这听起来很蠢，但大模型的推理机制本质上就是这样工作的——自回归生成，每次预测下一个 token，都要"看"一遍之前所有的 token。

如果你真的每次都重新算，那代价就太大了。假设上下文有 4096 个 token，每次都要做 4096 次注意力运算，这不是在烧 GPU，这是在烧钱。

所以聪明人就想了：把每次计算出来的中间结果（Key 和 Value）缓存下来，下次直接查表。这就是 KV Cache——一张专门给 Attention 用的草稿纸，写过的内容不用重写。

草稿纸越写越大 ​

问题来了。这张草稿纸是画在显存里的，而且随着对话变长，它会越来越大。你生成第一个 token 时，草稿纸只有几个字；生成到第 1000 个 token 时，草稿纸上已经有 1000 组 Key-Value 了。这些数据全得存在显存里，而且是以 O(n) 的速度线性增长的。

一个粗略的数字：70B 参数的模型，每个 token 的 K 和 V 向量大概占用 1MB 显存。听起来不大？但 4096 个 token 就是 4GB，16384 个 token（大约 1 万字）就奔着 16GB 去了——要知道很多显卡的显存总共也就 24GB、48GB。

这就是 KV Cache 的核心矛盾：显存有限，上下文长度无限。

PagedAttention：像操作系统一样"分页" ​

固定座位的问题 ​

传统方法是怎么管理 KV Cache 的呢？打个比方你就明白了。

想象你去餐厅吃饭，服务员告诉你："不好意思，您这个团体必须坐连续的空位，不能分开坐。"结果你们 10 个人，但能放 10 把连续椅子的区域只剩 8 个位置，另外一个区域有 5 个位置但中间有根柱子。你们宁愿换一家餐厅，也坐不下。

这就是传统 KV Cache 的问题——预先分配固定大小的连续内存块。每个 token 的 K/V 向量组被当成一个整体，必须放在显存的一块连续区域里。一旦碎片化，明明还有足够的空闲显存，但就是凑不出"连续"的空间来。

结果就是：显存利用率只有 20%~40%，一大半空间都浪费在"等待连续座位"上了。

分页管理：让碎片化不再成为问题 ​

vLLM 团队从操作系统那里学来了一招：分页管理（Paging）。

在操作系统里，物理内存是按固定大小的页（Page）来管理的。一个进程可以占用很多不连续的物理页，但操作系统维护一张页表，把虚拟地址映射到物理地址。对 KV Cache 来说，vLLM 也做了类似的事情：

Block as Page：把显存分成固定大小的"页"（比如 16 个 token 为一页），不同的请求可以共享不同的页，不必连续。

虚拟地址空间：每个请求有自己的虚拟 KV 缓存区，看起来是连续的，但从物理上看，数据散落在显存各处，按需分配。

Copy-on-Write：父子请求可以共享只读页面，只有在修改时才复制（这在 Beam Search 等场景下特别有用）。

你可以把 PagedAttention 理解为：不再要求"一整排连续座位"，而是接受"分散就座"，服务生靠手里的座位图（页表）知道你们都坐在哪。

vLLM 用了 PagedAttention 之后，显存利用率从 ~20% 提升到了 ~80%，同等显存下能服务的并发请求数量翻了好几倍。这就是为什么现在很多人部署大模型，首选就是 vLLM。

MLA：把"笔记"压缩成"关键词" ​

DeepSeek 的思路 ​

DeepSeek-V2 提出了一个不同的优化思路，叫 Multi-head Latent Attention（MLA）。它不是在管理方式上做文章，而是在"存储什么"上做文章。

先想一个场景：你有一个笔记本，里面密密麻麻记了 1 万字的会议记录。但你发现，每次需要参考某段内容时，你并不是真的要把那一万字全部再看一遍，你只需要一些关键要点就够了。

MLA 就是这个思路。它不直接缓存原始的 Key-Value，而是先把它们压缩到一个低维空间，只存那个压缩后的"摘要"，计算注意力时再临时"解压"出来。

低秩分解：抽象画比写实画省空间 ​

数学上这叫 低秩分解（Low-Rank Factorization）。说起来吓人，但类比一下很好懂。

你有一张 512×512 的人脸照片（密密麻麻的像素点），我让你只记住 64 个特征来描述这张脸：眼睛间距、鼻子长度、颧骨高度……你压缩到 64 个数，再看到这些数时，虽然不是原图，但能大概"重建"出这张脸的感觉——这就是低秩压缩。

MLA 的操作类似：原始的 K 矩阵是 [seq_len, num_heads, head_dim]，压缩后变成了一个很小的"潜在向量" c_t^{KV}，存的是 num_heads × head_dim 降维到 low_rank_dim 的结果。

推理时怎么用？每个 token 只需要解码一个额外的"位置 key"，结合缓存的低秩向量，就能重建出完整的注意力结果。

省了多少？ ​

粗略估算：MLA 每个 token 只存一个低秩向量 + 解码用的 key，相比原始的完整 K 矩阵，KV Cache 大小减少了约 5~8 倍。

更妙的是，这压缩过程是端到端可学习的——模型在训练时就学会了"什么样的压缩方式保留了关键信息"，不是简单丢弃数据，而是真的学到了一个更好的表示。

除了 PagedAttention 和 MLA，还有什么招？ ​

其实这两条路代表了两个优化方向，业界还有很多有意思的实践：

StreamingLLM 的"锚点策略"：超长对话没法把全部历史都缓存，StreamingLLM 发现保留开头几个 token（汇聚注意力）和最近几个 token，中间用滑动窗口——这个组合居然效果出奇地好，被形象地称为"注意力汇聚（Attention Sink）"。

MQA/GQA：减少"头数"：Multi-Query Attention 让所有注意力头共享一份 K/V，Multi-Head Attention 有 N 个头就有 N 份 K/V，数据量差 N 倍。GQA（Grouped-Query Attention）是个折中方案，N 个 Query 头分成 G 组，每组共享一套 K/V。

H2O（Heavy-Hitter Oracle）：基于一个观察——有些 token 对最终结果影响巨大（比如关键决策词），有些是"凑数的"。H2O 选择性地只保留那些高频出现在注意力加权里的 Heavy-Hitter，把其他的丢弃掉。

怎么选？ ​

说了这么多，你可能会问：那我实际用的时候该关注什么？

简单总结一张表：

场景推荐方案原因
高并发部署，多用户同时请求PagedAttention（vLLM）显存利用率翻倍，并发量大幅提升
超长上下文，32K+ tokenStreamingLLM / 滑动窗口全量缓存不现实，必须选择性保留
极致压缩，想省更多显存MLA / GQA从存储结构上动手，从根本上减少体积
对延迟敏感，要求实时响应MQA减少内存访问，延迟明显降低

写在最后 ​

回过头来看，KV Cache 优化的故事本质上是一个工程哲学的较量：

PagedAttention 是外部优化——物理显存就那么多，我通过更聪明的管理方式把碎片化消灭掉，让每一字节都物尽其用。

MLA 是内部优化——数据本身有冗余，我从根本上压缩它，存一个"精华版"，用时再展开。

两条路没有绝对的好坏，更多是看你的场景侧重什么。如果你做的是高并发服务，PagedAttention 的收益是立竿见影的；如果你在做模型结构创新，MLA 的思路更值得借鉴。

显存永远是大模型推理的稀缺资源。理解它、管理它、压缩它——这才是让大模型真正走向实用的必经之路。
