OpenClaw Skills：让大模型"会使用工具" ​

上一篇文章我们聊了Agent——大模型怎么通过工具真正"干活"。今天我们把视角再收窄一点，专门讲讲Skill（技能）——这是让大模型用好工具的具体工程方法论。

打个比方：如果Agent是一家公司，Skill就是每个岗位的"工作手册"。手册写得好，新人来了就能上手；写得乱七八糟，再强的Agent也会抓瞎。

什么是Skill ​

Skill的本质是一组结构化的工具定义。它告诉大模型：

  • 我有什么能力（能做什么）
  • 怎么调用这些能力（什么情况下用，用什么参数）
  • 调用后会发生什么（返回什么结果）

你可以把Skill理解为API的"智能包装"。普通API是给人看的——你需要知道参数含义、会写调用代码。Skill是给大模型看的——大模型读了之后，自己能判断什么时候该用、该传什么参数、该怎么看结果。

举一个生活中的类比：

普通API像是药品说明书——字密密麻麻，告诉你"每次两片，饭后服用，禁忌如下"。你得自己读、自己理解、自己执行。

Skill像是药盒上的大字说明——"头痛？吃这个。一天一粒。"简短、直接、傻瓜化。大模型不需要是个医学专家，照着说明就能用对药。

Skill的四个组成部分 ​

一个完整的Skill通常包含四个部分：

1. 描述（Description） ​

这是最关键的部分——大模型靠它判断"什么时候该用这个Skill"。

描述要写清楚三件事：

  • 能做什么：核心能力是什么
  • 什么时候用：触发条件是什么，用户什么需求时会用到
  • 约束限制：有什么前提条件，有什么不能做的事

比如一个"发邮件"的Skill，描述可能是：

发送电子邮件技能。
触发场景：用户说"发邮件"、"给某人发消息"、"通知XXX"、提供收件人和正文内容时使用。
前提：必须有收件人和正文；不支持附件（需用其他Skill处理）。

这个描述就是给大模型看的，不是给人看的技术文档。要用自然语言、具体场景来写，让大模型一看就明白。

2. 参数定义（Parameters） ​

当大模型决定调用Skill时，它需要知道该填什么参数。参数定义就是这份"表格"。

好的参数定义要包含：

  • 参数名：简洁、语义清晰
  • 参数类型：字符串、数字、布尔值……
  • 是否必填：可选还是必须
  • 描述：这个参数是什么意思
  • 示例：给大模型一个参考值
参数：
- to（必填，字符串）：收件人邮箱地址，如 "lisi@example.com"
- subject（必填，字符串）：邮件主题，不超过100字符
- body（必填，字符串）：邮件正文内容
- cc（可选，字符串）：抄送地址，多个用逗号分隔

参数定义越具体，大模型填错的概率越低。特别要注意类型约束——如果参数应该是数字，但大模型填了个"三"，系统就能提前发现错误。

3. 执行逻辑（Implementation） ​

这就是实际的代码——收到调用请求后，Skill干什么。

执行逻辑通常是一个函数或一个脚本，输入是参数，输出是结果。

不过值得注意的是：Skill的执行逻辑不一定要大模型自己写。实际上，很多Skill的执行逻辑是提前写好的模板，大模型只负责决定调用哪个Skill、填什么参数。这和Agent的"大模型做决策，工具做执行"是一脉相承的。

打个比方：Skill的执行逻辑像是餐厅厨房里的标准流程——"收到酸辣土豆丝的订单→土豆切丝→热油→下锅→调味→装盘"。大模型不需要会炒菜，它只需要知道"这个人点了酸辣土豆丝，应该走这个流程"。

4. 结果处理（Response） ​

Skill执行完了，要把结果告诉大模型。结果怎么返回、格式是什么，很大程度上影响了大模型下一步的决策质量。

好的结果格式应该：

  • 结构化：让大模型容易解析，不要返回纯文本"操作成功"
  • 有状态：告诉大模型是成功还是失败
  • 有信息量：成功时返回有意义的数据，失败时说明原因和建议
{
  "status": "success",
  "message": "邮件已发送至 lisi@example.com",
  "details": {
    "message_id": "abc123",
    "sent_at": "2026-07-16T10:00:00Z"
  }
}

vs 差的返回：

"邮件已发送"

前者让大模型知道邮件ID、发送时间，如果需要后续操作（比如"取消这封邮件"），大模型有据可查；后者只是一个状态，大模型只知道"发了"，后续什么也做不了。

Skill生态：让工具真正可复用 ​

单个Skill的价值有限，真正厉害的是Skill生态——一堆Skill放在一起，大模型可以自由组合使用。

想象一下你的工作台上有这些Skill：

  • 发邮件
  • 查日历
  • 创建会议
  • 搜索文档
  • 发送企业微信消息
  • 操作数据库
  • 生成Excel报表

当用户说"帮我把本周的销售数据整理成表格，发给老张"时，大模型自动拆解：

  • 查日历 → 确定本周是哪几天
  • 操作数据库 → 拉取本周销售数据
  • 生成Excel报表 → 整理数据成表格
  • 发邮件 → 把表格发给老张

这就是Skill生态的力量：单个Skill简单，但组合起来可以完成复杂任务。大模型不需要提前知道"怎么把查数据、发邮件串起来"，它只需要知道每个Skill是干什么的，执行顺序由任务目标自动推导。

这种模式在软件工程里有个类似概念——Unix哲学：每个工具只做一件事，但工具之间可以通过管道组合。Skill就是AI时代的"Unix工具"。

怎么写一个好的Skill ​

结合上文，说几个实战要点：

描述要像"用户故事"，不要像"技术文档"

❌ 差的描述：

本Skill提供SMTP邮件发送功能，支持HTML正文和附件……

✅ 好的描述：

发送邮件技能。当用户说"发邮件给XX"、"帮我通知XX"、"把XX发给他"时触发……

参数要"所见即所得"

参数名和描述要用用户熟悉的语言。大模型填参数时，会参考参数名和描述——如果参数名叫recipient_address，大模型可能会填"老张的邮箱"，而不是"lizhang@company.com"。

失败要给出建议

如果Skill执行失败了，返回的错误信息不应该只是"错误码E500"。大模型需要知道怎么修复。建议格式：

json
{
  "status": "error",
  "error": "收件人地址格式不正确",
  "suggestion": "请检查邮箱格式，确保包含@符号和域名"
}

边界情况要提前处理

Skill要在输入层面做严格校验——如果参数类型是邮箱，就提前校验格式，不要把错误传给大模型处理。大模型擅长推理，但不擅长格式校验，这种事应该交给代码。

前端视角：Skill在前端开发中的应用 ​

作为前端开发者，你可能不自己写底层的Skill执行逻辑，但你完全可以参与Skill的设计和集成：

设计前端产品的Skill体系：如果你的AI产品需要接入各种工具，你需要思考哪些Skill是必要的、它们之间的关系是什么、如何组织Skill的描述让模型最容易理解。

构建内部工具Skill：你团队内部的很多重复工作（发版通知、查监控数据、管理CI/CD）都可以包装成Skill，让AI助手帮你执行。

体验Skill生态：试试把几个Skill组合起来，让AI帮你完成一个需要多步骤的任务，体验一下"工具组合"的力量。

总结一下：Skill是让大模型真正用上工具的方法论——它用结构化的描述让大模型"认识"工具，用标准化的接口让大模型"调用"工具，用生态化的组合让大模型"玩转"工具。

掌握了Skill的设计思路，你就掌握了让AI从"能说"到"能做"的那把钥匙。
