KV Cache：大模型推理加速的"记忆复用术" ​

你有没有过这种经历——写一封很长的邮件，写到一半突然发现前面某句话有个词用错了？普通人会翻回去改，AI 大模型在推理的时候也面临类似的问题：每生成一个新 Token，要不要把之前算过的中间结果都扔掉？

答案显然是否定的——扔掉太浪费了。KV Cache 就是来解决这个问题的：把已经算好的 Key-Value 存下来，每次生成新 Token 时直接复用，而不是从头再算一遍。这篇文章来好好聊聊这个"记忆复用术"。

Attention 里的"中间商"：K 和 V 是什么？ ​

先快速回顾一下 Attention 机制中的 Q/K/V。

在 Transformer 的每一层里，输入序列的每个 Token 都会被三个权重矩阵分别投影成 Query（Q）、Key（K）、Value（V）三个向量。Attention 的计算简单说就是：用 Q 去"查询"每个 K，找到和它最相关的 V，然后加权求和得到输出。

Attention(Q, K, V) = softmax(QK^T / √d) × V

这里的 Q、K、V 都是中间产物。在标准推理过程中，每生成一个 Token，模型需要对整个序列（从第 1 个 Token 一直到现在）重新跑一遍 Attention——包括那些之前已经算过的 Token。每次生成都在"重读全文"，效率极低。

Cache 的直觉：别让算过的东西白算 ​

打个比方。你在做阅读理解题，第一遍读文章花了 5 分钟，做完第一题后做第二题，你会把文章扔掉从头读吗？不会的，你会边翻边回忆。

标准推理就是这样：每生成一个 Token，就相当于做一道新题，而"重新读全文"这个操作在计算上非常贵（O(n) 的 Attention 计算）。但其实在生成第 T+1 个 Token 时，前面 T 个 Token 的 K 和 V 完全不会变——它们只取决于输入，是"一次性"的投入。

KV Cache 的核心思想就是：把这些"一次性投入"存起来，第一次算完，后面直接查表。 这样每次生成新 Token 时，只需要计算当前 Token 的 Q（加上一个新的 K 和 V），然后去"查"之前缓存好的 K 和 V，Attention 计算就变成了 O(1) 而不是 O(n)。

显存：甜蜜的代价 ​

天下没有免费的午餐。KV Cache 虽然快，但存 K 和 V 需要显存（GPU Memory）。

算一笔账：假设模型有 L 层，隐藏维度是 d_model，每个 Token 产生的 K 和 V 各占 d_model 维的向量。如果序列长度是 8192（即能处理约 6000 中文字符），每层有 2 个注意力头（K 和 V），每个元素用 FP16（2 字节）存储：

显存占用 ≈ 2 × L × 8192 × d_model × 2 字节

对于一个 70B 参数的模型，层数约 80，d_model ≈ 8192，光 KV Cache 就能轻松吃掉几百 GB 显存。这直接导致能同时"处理"的请求数量大幅减少——显存被 Cache 塞满了，新请求只能排队。

所以 KV Cache 既是加速器，也是显存杀手。优化 KV Cache 就是优化推理系统的核心战场之一。

PagedAttention：像操作系统管理内存一样管理 Cache ​

既然 Cache 要占用显存，那能不能像操作系统管理内存一样，用"分页"的方式管理它？

传统做法是把整个序列的 KV 连续存储在一大块显存里。这有什么问题呢？想象你同时处理 10 个请求，它们的序列长度各不相同——短的占不满，长的要扩展，连续存储会产生大量碎片。更要命的是，当一个请求生成完毕，你没法精准释放"中间的一段"，只能把整个 Cache 扔掉。

PagedAttention 是 vLLM 提出的解决方案，灵感来自操作系统的虚拟内存分页机制。它把 KV Cache 划分成固定大小的"页"（Page），每个请求不需要一整块连续显存，而是按需分配物理页。这样：

  1. 显存碎片大幅减少，利用率大幅提升
  2. 多个请求的 Cache 可以交叉存放，像搭积木一样紧凑
  3. 请求结束后，按页释放，不像以前那样粗粒度地扔掉整块

结果是：同等显存下，vLLM 可以支持大约 2-4 倍的并发请求数，同时吞吐量提升数倍。这也是为什么现在主流的推理服务（尤其是做批量推理的）几乎都在用 PagedAttention 相关的技术。

MLA：DeepSeek 的"压缩记忆"思路 ​

除了在 Cache 管理层面做优化，还可以在"存什么"上动脑筋。

DeepSeek-V2 提出的 Multi-head Latent Attention（MLA） 就是一种更激进的思路：与其存储完整的 K 和 V，不如对它们做低秩压缩（Low-Rank Compression），用一个更小的隐向量来表征。

这就好比：你不需要记住一整本书的每一个字，只需要记住每章的摘要就够了。MLA 把 K 和 V 压缩到 latent space 里，虽然理论上信息有损失，但实践中发现对模型效果的影响微乎其微，而显存节省却非常可观。

当然，压缩是有代价的——解压缩需要额外的计算步骤。MLA 的设计是让压缩带来的显存节省远大于解压缩的成本，最终达到"省显存且不太影响速度"的效果。

投机解码（Speculative Decoding）：多猜几步一起验证 ​

还有一个和 KV Cache 紧密相关的推理优化叫"投机解码"，它虽然不直接优化 Cache，但和 Cache 有协同效应。

传统解码是一个 Token 一个 Token 地往外蹦——每个 Token 都得等前一个算完才能开始算。投机解码的思路是：让一个小模型（Draft Model）快速"猜"出接下来 N 个 Token，然后用大模型一次性验证这 N 个。

如果大模型和小模型的"品味"差不多，N 个 Token 全部命中，那推理速度就是 N 倍提升。如果中间某个 Token 不对，大模型就接管，Cache 从"猜错"的 Token 位置重新开始。

这个策略的巧妙之处在于：验证 N 个 Token 的计算量，远小于从头生成 N 个 Token，因为 Cache 让验证步骤变得很便宜。就像你写邮件时，用自动补全快速敲完一段，再检查——改错字的代价远小于重写。

KV Cache 的实际影响 ​

说了这么多底层技术，你可能想知道：这东西到底影响什么？

对于普通用户来说，最直接的感知就是：响应速度和并发能力。没有 KV Cache，生成一段长文本可能需要几十秒；有了 Cache，同样的文本几秒就能出来。而 PagedAttention 这类技术，则决定了服务商能在同等硬件下服务多少用户——这直接影响 API 价格和可用性。

对于前端开发者来说，理解 KV Cache 有助于你在设计 AI 应用时做更好的预估：长对话的成本和延迟都不是线性增长的——前几个 Token 贵（需要计算和存储），后面的 Token 便宜（直接复用 Cache）。所以一个 2000 Token 的回答不是"1000 Token 回答的两倍贵"，而是前半段贵，后半段便宜很多。

一句话总结 ​

KV Cache 就是大模型推理时的"记忆复用术"——把 Attention 层已经算好的 Key-Value 存起来，避免每次生成新 Token 时重复计算。代价是占用显存，解决方案是 PagedAttention 的分页管理，或者 MLA 的低秩压缩。配合投机解码，推理速度还能再上一个台阶。

了解这个机制，下次你跟 AI 对话时脑子里就有画面了：模型不是在"重新读你说过的话"，而是在翻一本已经划好重点的笔记。
