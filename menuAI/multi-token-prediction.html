MTP：一次"猜"多个词的新范式 ​

传统大模型一次只猜一个词，Meta 的新研究说：能不能一次猜多个？猜对了加速，猜错了也没关系。这套方法叫做 Multi-Token Prediction，简称 MTP。

开头：为什么大模型"一个字一个字往外蹦"很慢 ​

用过 ChatGPT 或者 Claude 的同学，应该都注意到了一个现象：输出的文字是逐渐显现的，一个字一个字、一句话一句话地往外"吐"。

这背后不是打字员在打字，而是大模型的自回归生成机制在起作用——每生成一个新 token，都要完整跑一遍整个模型的计算。模型越大、生成的文本越长，这个过程就越慢。

打个比方：想象你在做一个翻译题，传统的方式是遮住一整句话，只露出一个词，让你猜这个词是什么。猜完后再遮住下一个，继续猜。这当然准确，但效率很低——因为每次你其实都读懂了整句话的大部分内容，却只能猜一个词。

MTP 的核心思想：能不能一次多猜几个？ 就像把遮住的词从1个变成3个、4个，让你一次预测多个位置。

投机解码：先猜再验证 ​

在说 MTP 之前，先介绍一个更直观的概念——投机解码（Speculative Decoding），它和 MTP 其实是"同一个思想的两个版本"。

猜测-验证框架 ​

投机解码的思路分两步：

  1. 快速猜测：用一个小的"草稿模型"（Draft Model），一次快速生成多个候选词
  2. 批量验证：用大模型（Target Model）一次验证这些候选词——如果验证通过，就一次性"接受"这一串词；如果某个词不通过，就从那个位置开始重新猜

你可以理解为：助理先帮你写了一句话，你只需要快速读一遍确认没问题就通过了，不用自己重写每个字。

text
传统方式（Target Model 单独跑）：
输入 → 计算1 → 词1 → 计算2 → 词2 → 计算3 → 词3 → 计算4 → 词4 → ...
（每步都要完整计算，慢）

投机解码：
草稿模型快速猜：词1 → 词2 → 词3 → 词4（几个词一次猜出来）
Target Model 验证：[词1 ✓][词2 ✓][词3 ✗] → 停止，从词3重新开始
→ 一次性接受了词1和词2，省去了2次完整计算

为什么这个方法work？ 核心假设是：草稿模型在大多时候是对的。因为语言生成有局部规律性（"今天天"后面大概率是"气"，"你好"后面大概率是"啊"），小模型也能猜对这些"简单"的部分。只有遇到复杂推理、罕见知识时才出错，这时候再用大模型来"兜底"。

实际效果：通常能实现 2~4倍推理加速，且输出质量和直接用大模型完全一致（因为验证的是大模型的判断）。

MTP：把"猜测"内置到模型里 ​

投机解码需要两个模型（一大一小），部署起来比较麻烦。Multi-Token Prediction（MTP） 的做法更优雅：把"一次猜多个词"这件事，直接训练进同一个模型里。

训练目标：从"猜1个"到"猜多个" ​

传统语言模型的训练目标是：给定前N个词，预测第N+1个词。

MTP 的训练目标是：给定前N个词，同时预测第N+1、N+2、N+3...个词，用多个独立的"预测头（Predictor Head）"分别负责不同位置的预测。

你可以理解为：不是派一个人每次猜一个词，而是同时派3个人，每个人负责猜不同位置的词。第1个人猜下一个词，第2个人猜下两个词，第3个人猜下三个词——他们的预测结果会互相作为参考。

text
传统目标：预测 Token[t+1]
MTP 目标：同时预测 Token[t+1], Token[t+2], Token[t+3], ...
         ↓
    训练3个独立的预测头

MTP 的两种主流实现：Vanilla vs Eagle ​

MTP 不是一种固定算法，目前主流有两条技术路线：Vanilla MTP 和 Eagle（有时也叫 MTP-Eagle）。

Vanilla MTP：最直觉的做法 ​

Vanilla MTP 的预测头是串行连接的：

text
输入 → 主模型 → 预测头1（猜t+1）→ 预测头2（基于头1的结果猜t+2）→ 预测头3（基于头2的结果猜t+3）

这和自回归生成的顺序一样，只不过把计算"压缩"了——每次不是跑完整的主模型，而是只跑预测头。

优点：结构简单，容易理解 缺点：预测头之间有依赖关系，t+3 的预测依赖 t+1 和 t+2 的结果，一旦前面错了后面也会错（误差累积）

Eagle（MTP-Eagle）：并行猜测，更激进 ​

Eagle 的核心改进是——让所有预测头并行工作，不依赖彼此的结果。

怎么做到的？它用的是更深的预测头结构，让每个预测头自己就能"看到"足够的信息来做预测，而不是依赖前一个预测头的输出。

text
输入 → 主模型 → 预测头1（猜t+1） 预测头2（猜t+2） 预测头3（猜t+3）
                ↑            ↑            ↑
              都从主模型的隐藏状态获取信息，独立预测

为什么 Eagle 更快？ 因为推理时，Eagle 的每个预测头可以同时运行（并行），不需要等前一个头完成。这比 Vanilla 的串行结构理论上快了接近3倍。

DeepSeek-V3 采用的就是 Eagle 风格的 MTP 架构，所以它成为了第一个在生产环境大规模使用 MTP 的主流模型。

MTP 在实际推理中怎么用 ​

MTP 的训练目标变了，但推理时的用法才是关键。目前有三种使用方式：

方式一：完整 MTP 推理（最激进） ​

推理时，模型同时用所有预测头来生成，每次生成多个 token。这需要推理框架（如 vLLM、SGLang）做特殊适配。

理论加速：3~4倍（取决于预测头数量） 代价：实现复杂度高，需要框架支持

方式二：投机解码（保守但实用） ​

训练一个 MTP 模型，用它的预测头来辅助一个更大的模型做投机解码。这是最常见的部署方式——不需要改动推理框架，只需要把 MTP 模型当"草稿模型"用。

DeepSeek-V3 就是这样用的：MTP 预测头作为"小模型"加速主模型的推理。

方式三：辅助训练目标（最常用） ​

很多模型在训练时会加入 MTP 作为辅助损失函数，帮助模型学得更好，但推理时仍然用传统自回归。这种方式不会加速推理，但能提升模型质量。

打个比方：就像学生做题时用"一题多解"来加深理解（MTP 训练），但考试时还是用标准方法答题（推理时不依赖 MTP）。

MTP 为什么有效：几个关键原因 ​

1. 更丰富的训练信号 ​

传统训练每个 token 只有一个标签（对或错）。MTP 提供了多个预测目标，每个预测头都在独立学习"下一个位置应该是什么"。

类比：传统训练像是每次只练习一个填空题；MTP 像是一次性做3道关联的填空题，你对全局的理解会更深。

2. 强迫模型做"规划" ​

当预测头需要预测 t+3 的位置时，它必须理解"如果要生成到这个位置，前面几个词应该怎么排列"。这逼迫模型在生成之前就在内部做某种"规划"。

3. 减少"重复生成"问题 ​

传统模型在生成长文本时容易陷入重复循环（"今天天气很好，天气很好..."）。MTP 因为有多个预测头同时工作，单个预测头的局部重复会被其他预测头"纠正"，整体上减少了这个现象。

局限性：MTP 不是银弹 ​

说了这么多 MTP 的好处，也得聊聊它的局限：

1. 预测头数量有上限

预测头不是越多越好。目前主流是2~4个头，超过这个范围收益递减，而且每个头都有参数量，增加太多会让模型变大。

2. 预测头之间存在一定误差累积

即使 Eagle 用并行设计，预测头之间仍然有相关性。当一个位置预测错误时，后续预测的准确率会下降。这也是为什么大多数系统仍然需要在"验证"步骤中用大模型兜底。

3. 对短文本没有明显优势

如果生成的文本只有几个词（比如回答一个"是/否"问题），MTP 的开销反而比传统方法高。

总结：一次多猜几个，是一种思维方式 ​

MTP 的本质，是对"生成效率"这个问题的一次范式级改进：

  • 投机解码 = 外部加一个小模型来猜
  • Vanilla MTP = 把猜测能力内嵌到模型里，但预测头串行工作
  • Eagle MTP = 把猜测能力内嵌，且预测头并行工作，效率最高

DeepSeek-V3 的实践已经证明：MTP 在生产环境是可行的，而且确实能带来 1.8~3 倍的真实加速。

对于前端同学来说，理解 MTP 的意义不在于自己实现它，而在于理解大模型推理加速的思路：不是让大模型跑得更快，而是用小模型替代大模型做它能做的事，把大模型留给真正需要它的时刻。这个思路在未来会越来越常见。
