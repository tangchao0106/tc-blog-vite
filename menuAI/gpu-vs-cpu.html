GPU vs CPU：为什么大模型"吃"显卡 ​

如果你用过 ChatGPT 或者部署过开源大模型，你大概率见过"NVIDIA A100""H100"这样的词。这些不是普通电脑的零件，而是专门的 GPU——显卡。奇怪的是，大模型为什么非要吃显卡？CPU 不行吗？

这篇文章就来把这个事讲清楚。

先打个比方 ​

想象你要做一道复杂的数学题：

CPU 像一个数学博士，拿到题目后，一步一步逻辑严密地推导，最终算出精确答案。优点是聪明、精准；缺点是一次只处理一个问题，解决大问题要花很长时间。

GPU 像一个大型小学数学班，里面有几百个小学生。每个人只会简单的加减乘除，但他们可以同时算。当你要处理海量的简单计算时（比如算 100 万道乘法题），一群小学生并行工作，比一个博士慢慢算快得多。

大模型训练和推理恰好就是这种"海量简单计算"——矩阵乘法、向量运算，全是小学生级别的基础数学。不是题目有多难，而是题目的数量太多。

CPU 和 GPU 的架构差异 ​

要理解为什么 GPU 更适合大模型，我们需要看一下两者的设计哲学。

CPU（Central Processing Unit，中央处理器）的设计目标是通用性和低延迟。它有几条特点：

  • 核数少：消费级 CPU 一般 4~16 核。Intel i9 是 24 核。每个核都很强，能处理复杂的指令分支（比如 if-else 逻辑）。
  • 大缓存：为了减少去内存读数据的时间，CPU 有大的一级、二级、三级缓存。类比的话，就像博士手边有一张很大的草稿纸，不用老跑图书馆。
  • 通用寄存器：能执行各种类型的指令，灵活但效率不一定最高。

GPU（Graphics Processing Unit，图形处理器）的设计目标是高吞吐量和并行度：

  • 核数极多：NVIDIA H100 有 16896 个 CUDA 核（流处理器）。即使是最强的消费级显卡 RTX 4090，也有 16384 个核。
  • 每个核相对简单：GPU 的核不需要处理复杂的分支逻辑，它擅长的是"大量同样的计算同时做"。就像 10000 个小学生同时算乘法，比 1 个博士串行算快。
  • 带宽高：HBM（High Bandwidth Memory）显存带宽达到 3.35 TB/s，而最快的 CPU 内存带宽也就几百 GB/s——差了 10 倍以上。

画个图帮助理解：想象一个工厂：

  • CPU 工厂：1 个高级技术工人，工厂里工具齐全，干什么都快，但一次只能干一件活。
  • GPU 工厂：10000 个流水线工人，每人只干一个简单动作，但 10000 件同样的活同时完成。

Transformer 为什么是"显卡友好型" ​

大模型基于的 Transformer 架构，其核心运算是矩阵乘法——本质上就是大规模并行加减乘除。

具体来说，Transformer 的 Self-Attention 模块中，有一个计算是 Q、K、V 三个矩阵相乘：Q × K^T × V。这个计算的特点是：

  • 高度并行：每个 token 和每个 token 的关系是独立计算的，天然适合并行。
  • 数据局部性差：Q、K、V 都是大矩阵，如果用 CPU 来算，要反复从内存里读数据，内存带宽会成为瓶颈。

GPU 的架构正好针对这两个特点做了优化：超多小核处理并行计算 + 超高显存带宽减少数据搬运延迟。

Tensor Core：专门为矩阵运算优化的"加速器" ​

普通的 GPU 核（CUDA Core）处理矩阵乘法已经比 CPU 快很多，但 NVIDIA 觉得还不够，于是在 Volta 架构之后引入了 Tensor Core——专门为矩阵运算设计的硬件单元。

一个 Tensor Core 一次能做 4×4 的矩阵乘法（FP16），一个 H100 有几百个 Tensor Core。算力差距有多大呢？

  • CPU（高端）：大约 1~2 TFLOPS（teraFLOPS，每秒万亿次浮点运算）
  • GPU CUDA Core（A100）：约 19.5 TFLOPS
  • GPU Tensor Core（A100）：约 312 TFLOPS
  • GPU Tensor Core（H100）：约 989 TFLOPS

差了 500~1000 倍。

这就解释了为什么大模型训练非 GPU 不可。用 CPU 训练 GPT-3 量级的模型，可能要几年；用 H100 集群，几周就够了。

显存（VRAM）：大模型的"工作台" ​

除了计算能力，还有一个关键资源是显存（VRAM）。

CPU 的"工作台"是内存（RAM），一般 16~128 GB。GPU 的显存 HBM 则从 16 GB（RTX 4090）到 80 GB（A100/H100）不等。

大模型的问题是：模型本身、输入数据、中间计算结果，都要放在显存里。

一个 7B 参数的模型，用 FP16 精度存储需要约 14 GB 显存——RTX 4090 的 24 GB 勉强能放进去，但如果加上推理时的 KV Cache、中间激活值，显存就捉襟见肘了。

  • 7B 模型：推理需要 ~14 GB（FP16），加上开销 ~20 GB+
  • 13B 模型：需要 ~26 GB（FP16），单卡放不下 A100/H100
  • 70B 模型：需要 ~140 GB，需要 2 张 H100（80 GB × 2）或更复杂的分布式方案

这就是为什么大模型推理需要多卡并行，或者用量化技术（INT8/INT4）来压缩模型体积——让模型能塞进显存。

前端开发者什么时候需要关心 GPU？ ​

你可能会想：我是前端开发者，日常写 React、调 CSS，为什么要知道这些？

1. 选择 API 服务时

当你选 OpenAI、Anthropic、DeepSeek 的 API 时，他们的计费差异背后有 GPU 成本。了解 GPU 的稀缺性和成本，能帮你理解为什么 AI API 比传统云服务贵这么多。

2. 本地部署模型

如果你想在本地跑个 7B 模型做demo，RTX 3060（12 GB）勉强能跑，RTX 4090（24 GB）体验好很多。知道显存需求能帮你选对硬件。

3. AI 产品性能调优

当你的 AI 产品响应慢，瓶颈可能在 GPU 排队、context 过长导致显存压力、或者模型过大需要量化。这些问题的诊断都需要一些 GPU 基础认知。

4. 理解 AI 新闻

以后看到"NVIDIA 发布新 GPU"、"H100 显存升级到 80GB"、"AI 训练成本下降 50%"这类新闻，你就知道背后是硬件进步在驱动算法进步——Scale 法则不只在算法层面，硬件也是 Scale 的核心维度。

总结 ​

大模型"吃"显卡的根本原因是：Transformer 的矩阵运算本质是大规模并行计算，GPU 的架构天然适配这种 workload。

  • CPU 像一个博士：通用、精准、但并行度低
  • GPU 像一群小学生：并行度极高，专为海量简单计算优化
  • Tensor Core 是"专项运动员"：矩阵运算比普通 CUDA 核快 10~50 倍
  • 显存是瓶颈：模型 + 中间状态都要放在显存里，16 GB 是本地推理的入门线

理解这些，下次当你听到"某某公司买了 10000 张 H100"、"AI 训练成本又降了"，你就能把这些新闻放到一个正确的技术背景里去理解了。

显卡不是魔法，它是把"大力出奇迹"这件事从理论变成现实的工程基础设施。
