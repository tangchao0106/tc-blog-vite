# 自回归生成：大模型为什么"一字一字往外蹦"

## 引子：那个让前端抓狂的"打字机"

用 ChatGPT、文心一言、DeepSeek 时，你一定见过这种画面：

```text
"今天天气很好，我"
"今天天气很好，我想"
"今天天气很好，我想去"
"今天天气很好，我想去爬"
"今天天气很好，我想去爬山"
...
```

模型像个"打字机"，**一次蹦一个字，蹦完一个再蹦下一个**。前端的流式输出（SSE）正是为这种现象设计的——它必须"等"模型"吐"完一个 token，才能渲染下一个。

为什么模型不能像前端一次性渲染整个列表那样，把整段话"啪"地一下全给你？

这就要从**自回归生成（Autoregressive Generation）** 的本质说起。

## 什么是自回归生成？

**自回归（Autoregressive, AR）** 听起来很唬人，拆开其实就两部分：

- **Auto** = 自己
- **Regressive** = 回归

合起来：**用自己之前的输出来预测自己的下一个输出**。

模型生成文本的过程，可以简化为这个循环：

```text
已知：context = "今天天气很好"
循环：
  1. 把 context 喂给模型
  2. 模型输出下一个 token 的概率分布
  3. 按某种策略选一个 token（比如概率最高的"想"）
  4. 把"想"拼到 context 后面
  5. 回到第 1 步
直到：模型输出结束符 <EOS>，或达到最大长度
```

**每一步的输入，都依赖上一步的输出。这就是"自回归"。**

数学上可以写成：

```
P(w1, w2, ..., wn) = ∏ P(wi | w1, w2, ..., wi-1)
```

整段文本的概率 = 每个 token 在已知上文条件下的概率之积。

## 训练时：自回归语言建模

训练 GPT 这类模型时，用的就是**自回归语言建模**（Autoregressive Language Modeling, ALM）：

```text
输入文本："今天 天气 很 好 我 想 去 爬山"
↓ 拆成 (输入, 标签) 对，每一对错开一位
("今天"        → 预测 "天气")
("今天 天气"   → 预测 "很")
("今天 天气 很" → 预测 "好")
("今天 天气 很 好" → 预测 "我")
...
```

注意**箭头方向**：从左到右，前面的 token 用来预测后面的 token。模型学到的就是"给定前文，下一个最可能是什么"。

这和"双向"模型（如 BERT）正好相反——BERT 是把整个句子灌进去，然后"挖掉"中间几个字让模型猜（下一篇我们要讲的**遮挡预测**）。

| 范式 | 代表 | 方向 | 任务 |
|------|------|------|------|
| 自回归 (AR) | GPT 系列 | 单向（左→右） | 预测下一个 token |
| 遮挡预测 (MLM) | BERT 系列 | 双向 | 猜被 [MASK] 掉的 token |
| 编码-解码 | T5 / BART | 双向编码 + 单向解码 | 翻译、摘要等 |

**GPT 走 AR，是因为它的目标是"生成"——必须一个接一个蹦；BERT 走 MLM，是因为它的目标是"理解"——只要挖空填对就行。**

## 训练 vs 推理：同源不同速

有个容易混淆的点：**训练时和推理时，模型做的事高度相似，但速度天差地别**。

| 阶段 | 输入 | 目标 | 并行度 | 速度 |
|------|------|------|:------:|------|
| 训练 | 完整句子 | 同时预测每个位置的下一个 token | 高（一次算完整个序列） | 快 |
| 推理 | 不断增长的 prefix | 一次只预测下一个 token | 低（必须串行） | 慢 |

训练时一句 1000 token 的话，模型可以**一次并行**算完——1000 个位置的"下一个 token 预测"同时进行。

但推理时，前面那个 token 不出来，后面没法开始——**你必须等模型"蹦"完第一个字，才能拿它和 prompt 一起喂回去算第二个字**。

这就是为什么前端流式输出要做成 SSE 协议——必须一点一点推，没法一次性"啪"地全给你。

## 一个最简的生成循环（Python）

我们用代码把上面的循环"写实"：

```python
import torch
from transformers import AutoModelForCausalLM, AutoTokenizer

model = AutoModelForCausalLM.from_pretrained("gpt2")
tokenizer = AutoTokenizer.from_pretrained("gpt2")

prompt = "今天天气很好，我"
input_ids = tokenizer.encode(prompt, return_tensors="pt")  # [1, 5]

max_new_tokens = 20
for step in range(max_new_tokens):
    # 1. 整个当前序列都喂给模型
    outputs = model(input_ids)

    # 2. 只取最后一个位置的 logits（下一步预测）
    next_token_logits = outputs.logits[:, -1, :]

    # 3. 简单策略：选概率最大的 token（贪心解码）
    next_token_id = torch.argmax(next_token_logits, dim=-1)

    # 4. 拼到序列末尾
    input_ids = torch.cat(
        [input_ids, next_token_id.unsqueeze(0)], dim=-1
    )

    # 5. 遇到结束符就停
    if next_token_id.item() == tokenizer.eos_token_id:
        break

print(tokenizer.decode(input_ids[0]))
# "今天天气很好，我想去爬山，顺便看看风景……"
```

注意循环里的**两个关键点**：

1. **每一步都重新把整段 `input_ids` 喂给模型**——是的，从第 1 步到第 20 步，序列越来越长，前面已经处理过的 token 反复被"算"了一遍。
2. **新生成的 token 又会成为下一次的输入**——这就是"自"回归，"自"己的输出回归到自己。

## KV Cache：别让模型做重复劳动

上面那个朴素循环有个巨大的浪费：算到第 6 步时，前面 5 个 token 的注意力计算已经做过一次了，第 6 步居然又把它们重新算了一遍。

**KV Cache** 就是为了解决这个浪费的：

- **K** = Key（注意力里的"键"）
- **V** = Value（注意力里的"值"）

模型每一层每一步都会算一组 KV。朴素做法是每步都重算所有 token 的 KV；KV Cache 的做法是**把历史的 KV 缓存下来**，新的一步只算新 token 的 KV。

```text
朴素做法（无 Cache）：
  Step 1: 算 token 1 的 KV
  Step 2: 算 token 1, 2 的 KV   (token 1 重算了)
  Step 3: 算 token 1, 2, 3 的 KV (token 1, 2 又重算了)
  ...
  Step N: 算 token 1..N 的 KV     (前面的全部重算 N 次)

KV Cache 做法：
  Step 1: 算 token 1 的 KV → 存起来
  Step 2: 只算 token 2 的 KV，复用 token 1 的 KV
  Step 3: 只算 token 3 的 KV，复用 token 1, 2 的 KV
  ...
  Step N: 只算 token N 的 KV，复用前 N-1 个
```

**计算量从 O(N²) 降到 O(N)，N 越长收益越大。**

这也是为什么 100K 上下文的大模型推理能"看起来不慢"——没有 KV Cache 早就慢成 PPT 了。

## 质量 vs 速度：鱼和熊掌

**优点：质量高**

- **全局一致性**：每一步都基于完整上文决策，模型能看到所有"已经说过的话"，逻辑连贯性天然好
- **训练简单**：目标函数清晰（就是预测下一个 token），端到端可微
- **Streaming 友好**：天然支持流式输出，前端用户体验好

**缺点：速度慢**

- **串行瓶颈**：第 N 个 token 必须等前 N-1 个算完，GPU 算力浪费严重
- **长文本退化**：序列越长，单步延迟累积越明显
- **不能并行生成**：没法像 BERT 那样一次性把整段话都"填"出来

一个直觉：

```text
BERT 类"遮挡预测"  → 像  填空题  ："今天____很好"，一次性填完所有空
自回归              → 像  写作文  ：必须先写"今天"，再写"天气"，再写"很"……
```

填空题显然快（一次性填完），但作文更连贯（因为每写一个字都考虑了"前面已经写了什么"）。

## 投机解码：让大模型"批改作业"

既然一个字一个字蹦太慢，能不能"猜"几步？

**投机解码（Speculative Decoding）** 是 2023 年 Google 和 DeepMind 几乎同时提出的加速方案，思路非常聪明：

1. **小模型（Draft Model）** 先一口气生成 K 个候选 token（快，但质量一般）
2. **大模型（Target Model）** 一次性"批改"这 K 个 token（一次前向传播搞定）
3. 候选 token 大模型认可的就保留，不认可的就重新采样

```text
小模型猜：今 天 天 气 很 好 我 想 去 爬 山      (5ms)
                       ↓ 一次性送大模型
大模型校验：✅今 ✅天 ✅天 ✅气 ✅很 ✅好 ✅想 ❌去 → 重采样
                       ↓
最终采纳 7 个 token，重新生成 1 个
总耗时：5ms(小) + 30ms(大) = 35ms
比自回归的 8 × 30ms = 240ms 快 6 倍
```

**关键点**：投机解码的输出分布和"纯自回归"严格一致，不会降低生成质量——这和"为了快而牺牲质量"的传统量化、剪枝完全不同。

类似的加速思路还有：

- **Medusa**：在模型最后一层加几个"预测头"，一次性预测多个位置
- **EAGLE / EAGLE-2**：用轻量级自回归模型做"特征级"的投机
- **Lookahead Decoding**：维护一个"候选池"，命中即用

## 前端开发者关心的事

知道自回归的原理后，前端在和大模型打交道时能少踩几个坑：

1. **流式输出是必须的**：SSE / WebSocket 推 token 是正确姿势，不要等模型生成完再一次性返回
2. **首字延迟（TTFT）比总时长更影响体验**：自回归的"串行"特性决定 TTFT 取决于 prompt 长度 + 首步推理时间，prompt 越短 TTFT 越短
3. **KV Cache 让长 prompt 也能"首字秒回"**：50K token 的 prompt 不会"卡死"，因为首步只算 prompt 一次
4. **投机解码对前端透明**：服务端用了什么加速方案，前端拿到的还是"一串 token 慢慢推出来"，实现细节不用关心

## 总结

> **自回归生成 = 用上一步的输出当下一步的输入，一字一字循环蹦出来。**
>
> - 训练时一次性算完，推理时必须串行——这是"慢"的根源
> - 全局一致性是它"质量高"的原因
> - KV Cache 缓解了重复计算，投机解码进一步突破串行瓶颈
> - 前端流式输出（SSE）天然适配这种"打字机"模式
