# 模型量化与剪枝：让大模型"瘦身"的两种思路

你有没有这种感觉：买了一件大衣，回家发现太大了，想改小一点，但又不希望改完之后穿着走形？

大模型的"瘦身"问题跟这很像。一个 70B 参数的模型，光是参数就占 140GB 内存，普通的 GPU 根本跑不动。但你又不能随便删参数，否则模型回答质量问题。

所以就有了两种主流的"改衣"技术：**量化（Quantization）** 和 **剪枝（Pruning）**。今天我们就来聊聊它们各自是什么、有什么区别、什么时候用哪个。

## 量化：把"精确数字"变成"简化代号"

### 一个生活中的类比

你去买水果，老板说"苹果 5 块一斤"。但如果他说"苹果 4.7 块一斤"，你大概率还是会当成"大约 5 块"来处理——因为精度到小数点后一位对你买水果这件事来说，没什么实际意义。

**量化就是这个思路**：把模型里的高精度数字（FP32、FP16）用更少的位数来表示，从而减少内存占用和计算量。4.7 近似为 5，就是一种"量化"。

### 具体怎么做的

模型参数在训练和推理时通常以 **FP32（32位浮点）** 存储，每个数字占 4 个字节。FP16 占 2 个字节，精度稍低。量化就是把这些 32 位或 16 位的数字"压缩"到 8 位（INT8）甚至 4 位（INT4）。

> 打个比方：原来每个数字用"厘米"来量（精度高，占空间），现在改成用"米"来量（精度低一些，但占用空间只有 1/100）。

常见的量化精度：

| 格式 | 位宽 | 内存占用（70B 模型） | 典型场景 |
|------|------|------------------|----------|
| FP32 | 32bit | ~280GB | 训练 |
| FP16/BF16 | 16bit | ~140GB | 推理基准 |
| INT8 | 8bit | ~70GB | 量化推理 |
| INT4 | 4bit | ~35GB | 极致压缩 |

INT4 听起来很诱人——内存直接降为原来的 1/8！但精度损失也会更明显，回答可能变得"不那么聪明"。

### 量化不是简单截断

很多人以为量化就是"把所有数字除以 2，然后取整"，其实不是这样。量化过程中最关键的一步叫**校准（Calibration）**：需要用一小批真实数据跑一遍模型，找到合适的缩放因子（scale），让量化后的模型在精度损失最小的情况下运行。

打个比方：你有一把尺子，刻度是厘米，现在想换成刻度为米的尺子。但直接换会有误差，所以你要先用一批样本量一量，找一个最优的换算比例，让误差最小。这个"找最优比例"的过程就是校准。

**主流的量化方法**：
- **GPTQ / AWQ**：后训练量化（PTQ），在模型训练完之后做，效率高，精度损失可接受
- **GGUF**：LLM.cpp 生态的量化格式，把量化参数和权重一起打包，CPU 也能跑
- **QAT（量化感知训练）**：在训练时就模拟量化效果，让模型提前适应"低精度"状态

### 量化后的模型能干什么

以 4-bit 量化的 Llama 3 70B 为例，原来需要 280GB 显存，现在只需要 35GB 左右，配上消费级双 RTX 3090 就能跑了。INT8 的版本大概需要 70GB，单卡 A100 80GB 刚好够。

这就是量化的价值：**让大模型从"只有大公司能跑"变成"科研团队也能玩得起"**。

## 剪枝：把"不重要的地方"删掉

### 另一个类比

你写一篇 5000 字的文章，回头看发现有两段话说的是同一个意思，删掉一段不影响阅读，反而更精简。大模型的剪枝就是这个思路：**找到那些"冗余"或"不重要"的神经元/连接，把它们删掉**。

### 剪枝的两种类型

**1. 权重剪枝（Weight Pruning）**
把那些数值接近零的连接直接删掉。直觉是：权重接近零意味着这个连接对最终结果影响很小，删了问题不大。就像你家的WiFi路由器，某个房间里信号特别弱（权重接近零），干脆就把那个节点删掉，不影响整体上网。

**2. 结构化剪枝（Structured Pruning）**
权重剪枝虽然有效，但删除的是细粒度的连接，导致模型参数变得稀疏（到处都是零零碎碎的空洞），硬件加速困难。结构化剪枝干脆把整个神经元、整个注意力头、整个层一起删掉，形成一个"天然紧凑"的小模型。

> 类比一下：权重剪枝像把一篇文章里每个句子里删掉几个不太重要的词，结果文章读起来断断续续；结构化剪枝像直接把某个段落删掉，剩下的段落完整连贯。

### 大模型剪枝的挑战

传统剪枝在 BERT 时代用得比较多，但大模型时代遇到了新问题：

- **大模型本身就是稀疏的**：训练好的大模型中，很多权重本来就接近零，剪枝空间有限。
- **难以判断"不重要"**：大模型的能力往往是 emergent（涌现）的，你以为某个连接不重要，删掉后可能影响某个意想不到的能力。
- **训练代价太高**：剪枝后通常还需要微调来恢复性能，但大模型微调成本极高。

所以目前主流的做法是**稀疏化训练**，而不是训练完再剪枝。即在训练时就施加正则化，鼓励模型天然形成稀疏结构，而不是训练完了再去删。

## 量化和剪枝：选哪个？

| | 量化 | 剪枝 |
|---|---|---|
| 原理 | 降低数字精度 | 删除不重要权重 |
| 内存影响 | 直接减少 | 部分减少 |
| 计算速度 | 显著提升 | 部分提升（需硬件支持稀疏计算） |
| 精度损失 | 取决于位数 | 取决于剪枝比例 |
| 工程成熟度 | 高（GPTQ、GGUF 广泛使用） | 中（结构化剪枝进展较快） |
| 对硬件要求 | 降低（INT8/INT4 有硬件加速） | 需要稀疏计算支持 |

**实践中，量化用得更多**，因为：
1. 工程实现成熟，工具链完善
2. 精度损失可预期、可控
3. 几乎所有硬件都对低精度计算有加速

剪枝更适合当你需要**彻底改变模型结构**（比如从 70B 变成 7B 这种极端情况）时，配合微调来用。

## 组合使用：1+1 > 2

有趣的是，量化和剪枝可以叠加使用。先剪枝让模型变小，再量化把剩下的参数压缩到低位宽。很多开源社区的"小模型"就是这么来的——比如 Llama 3 70B 经过剪枝+INT4量化，可以压缩到只有 4GB，在手机端跑都勉强能用了。

> 把这个过程想成装修：先拆掉不承重的墙（剪枝），再把剩下的家具缩小比例（量化）。最后原来的大三居变成了精装修的一居室，功能够用，但住起来完全没问题。

## 写在最后

理解量化和剪枝，有助于你理解为什么现在有那么多"小模型"可以选择。Qwen2 7B、Phi-3 Mini 这些小模型，很多都是在更大的模型上经过量化或剪枝得到的。

下次你在选模型时，如果看到 "INT4量化版" 或 "4-bit Q5_K_M" 这样的标注，就知道背后的技术原理了——模型被"压缩"过了，但核心能力还在。
