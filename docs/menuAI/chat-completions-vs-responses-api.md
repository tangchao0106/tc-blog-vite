# Chat Completions vs Responses API：大模型API的"两代同堂"

如果你用过 OpenAI 的 API，大概率是从 `chat/completions` 开始的。这是 2023 年发布的第一代 API，几乎所有大模型厂商都在模仿它—— Anthropic 的 Messages API、国内各家的对话接口，骨子里都是 Chat Completions 的设计。

但在 2025 年，OpenAI 推出了全新的 **Responses API**。两套 API 同时存在，官方也没说要淘汰旧的那套。于是很多开发者困惑了：我该用哪个？它们到底有什么区别？

今天我们就把这两代 API 的设计哲学掰开揉碎讲清楚，帮你做出选型决策。

## 先搞明白：Chat Completions 是什么？

Chat Completions 是"对话式"的 API。它的核心模型很简单：

**你发一段对话历史，模型回复一段新消息。**

打个比方：就像微信聊天——你把之前的聊天记录截图发给对方，对方看了一眼，接着往下聊。

典型的请求结构长这样（示意，不贴完整代码）：

`messages: [{role: "user", content: "你好"}, {role: "assistant", content: "你好呀"}, {role: "user", content: "讲个笑话"}]`

核心特征：

1. **无状态**：每次请求都是独立的，服务端不记住你是谁、之前聊了什么。你得自己把对话历史全带上
2. **纯文本对话**：本质上就是"你一句我一句"的消息序列
3. **单轮响应**：发一次请求，收一次回复，结束

说白了，Chat Completions 就是个**高级的聊天机器人接口**。它假设你只是在"聊天"——没有工具调用、没有多步任务、没有中间状态管理。

## Responses API：不是"聊天"，是"任务"

Responses API 的设计哲学完全不同。它不是围绕"对话"设计的，而是围绕**"任务"**设计的。

你可以理解为：Chat Completions 是"聊天窗口"，Responses API 是"项目管理器"。

聊天窗口的特点是——说一句回一句，说完就完了。项目管理器的特点是——一个任务可能需要多步操作、调用不同工具、中间有状态变更、最终交付一个完整结果。

举个具体的例子：

**用 Chat Completions 查天气**：你问"北京今天天气怎么样"，模型可能说"我没有实时天气数据"，或者瞎编一个。你得自己想办法——先调用天气 API 拿到数据，再喂给模型让它帮你总结。

**用 Responses API 查天气**：你定义一个任务"查北京天气"，模型自己判断需要调用天气工具，它去查了，拿到数据，帮你总结，一次请求搞定。

这个区别的本质是：**谁负责编排多步流程**——是你（开发者）手动拼，还是模型自己串？

## 五个核心区别，逐个拆解

### 1. 状态管理：自带 vs 自扛

Chat Completions 是无状态的——每次请求你得把整个对话历史带上。如果对话有 50 轮，每轮 500 个 Token，那你每次请求就要发 25000 个 Token 的历史。不仅浪费带宽，还浪费 Token 计费。

Responses API 支持**服务端状态存储**。你可以创建一个 Response 对象，后续请求只需要引用这个对象的 ID，服务端自动带上之前的上下文。

打个比方：Chat Completions 像你每次去超市都要把购物清单从头到尾念一遍；Responses API 像超市给你一个购物车——你往里加东西就行，结账时看车里有啥。

### 2. 工具调用：手动拼装 vs 内置编排

这是最实用的区别。

Chat Completions 的 Function Calling（函数调用）是这样的：

1. 你发请求，告诉模型有哪些工具可用
2. 模型回复一个"我想调用某个工具"的指令
3. 你去调用那个工具，拿到结果
4. 把结果作为新消息加到对话历史里
5. 再发一次请求，让模型根据工具结果生成最终回复

**4 步！**而且每一步都是你自己编排的——你写代码处理工具调用、拼接消息、管理状态。

Responses API 的做法：

1. 你发一个任务请求，告诉模型有哪些工具可用
2. 模型自己决定调用哪个工具、调用几次、按什么顺序
3. 一次请求返回最终结果

**1 步！**模型在内部自动完成了整个编排流程。

打个比方：Chat Completions 是"你当项目经理，AI 当执行者"——你分配任务，AI 做一步，你接结果，你再分配下一步。Responses API 是"你当甲方，AI 当全包团队"——你提需求，AI 自己搞定所有中间环节。

### 3. 多模态输入：拼接 vs 原生

Chat Completions 支持多模态（图片、文件等），但方式比较笨拙——你得在消息里手动构造 content 数组，把文字和图片分开描述。

Responses API 的多模态是**原生设计**的。你可以直接把文件传给 API，它内部处理编码、理解内容、整合到推理流程中。甚至还内置了文件搜索功能——你上传一个 PDF，模型可以直接在里面检索相关信息。

你可以理解为：Chat Completions 需要你先把食材切好装盘再给厨师；Responses API 你直接把食材丢给厨房，厨师自己处理。

### 4. 流式输出：两者都支持，但体验不同

两个 API 都支持 streaming（流式输出）。但 Responses API 的流式信息更丰富——它不仅输出文本 Token，还会流式推送工具调用状态、中间思考过程等事件。

Chat Completions 的流式就是"一个字一个字蹦出来"。Responses API 的流式更像"看直播"——你能看到主播在想什么、在用什么道具、在做什么操作，不只是最终的台词。

这对前端开发很重要：你可以根据流式事件实时更新 UI——显示"正在搜索文档……""正在调用工具……""正在分析结果……"，而不是只看到一个空白的 loading 状态。

### 5. 错误处理和重试：粗粒度 vs 细粒度

Chat Completions 的错误处理很简单——要么成功，要么失败。失败了你就重新发一次请求。

Responses API 因为是多步骤的，每个步骤都可能出问题——工具调用失败、搜索没找到结果、某一步推理出错。API 会返回更详细的错误信息，告诉你具体哪一步出了问题，你可以选择重试特定步骤而不是整个流程。

打个比方：Chat Completions 报错就像"手术失败，重来"；Responses API 报错像"手术中第三步出了状况，只需要修正第三步"。

## 一张对比表看全貌

想象一张对比表，左边是 Chat Completions，右边是 Responses API：

| 维度 | Chat Completions | Responses API |
|------|-----------------|---------------|
| 核心模型 | 对话（消息序列） | 任务（目标+编排） |
| 状态管理 | 无状态，开发者自扛 | 服务端存储，引用 ID |
| 工具调用 | 手动 4 步编排 | 模型自动编排 |
| 多模态 | 手动构造 content | 原生支持+内置搜索 |
| 流式输出 | 只输出文本 Token | 文本+工具状态+思考过程 |
| 错误粒度 | 整体成功/失败 | 分步骤状态和错误 |
| 适用场景 | 简单对话、单轮问答 | 多步任务、工具编排、复杂工作流 |

## 选型指南：什么时候用哪个？

这不是"新的一定比旧的好"——两个 API 各有各的甜点区。

### 用 Chat Completions 的场景

1. **简单对话应用**：聊天机器人、FAQ 问答、客服对话——你只需要"你一句我一句"，不需要复杂编排
2. **你已经有了成熟的代码**：如果你的项目已经在 Chat Completions 上跑得很好，迁移成本不值得
3. **需要最大兼容性**：几乎所有第三方库、教程、示例都基于 Chat Completions，生态最成熟
4. **你想要精确控制**：每一步工具调用你都想自己编排，不希望模型"擅自做决定"

### 用 Responses API 的场景

1. **多步任务编排**：一个用户请求需要调用多个工具、多轮推理才能完成——比如"帮我查天气然后推荐穿衣方案再写一段出行建议"
2. **Agent 类应用**：你的 AI 需要自主决策、自动搜索、调用工具——Agent 天然更适合 Responses API
3. **需要服务端状态**：长对话、多轮交互，不想每次都把全部历史带上
4. **文件密集型任务**：需要上传文档、搜索文档、基于文档内容回答问题
5. **你想要更好的 UX**：流式展示工具调用过程，让用户看到 AI "正在做什么"

打个比方：Chat Completions 是手动挡汽车——简单场景下操控感好，但复杂路况下操作繁琐。Responses API 是自动挡——日常驾驶更轻松，但赛车手可能还是想用手动挡的精确控制。

## 实际迁移的注意事项

如果你打算从 Chat Completions 迁移到 Responses API，几个实操要点：

**不是逐行替换**：两个 API 的请求/响应结构差异很大，不是换个 endpoint 就行。你需要重新设计请求逻辑，尤其是工具调用部分。

**先并行运行**：不要一刀切切换。可以先在新功能上用 Responses API，旧功能继续用 Chat Completions，逐步迁移。

**关注 Token 成本**：Responses API 让模型自动编排工具调用，这意味着模型可能"想得多"——思考过程也消耗 Token。监控一下成本变化。

**前端要做适配**：流式事件的类型变了，你的 SSE 解析逻辑需要更新。好消息是 Responses API 的流式信息更丰富，你可以做更好的 UI 反馈。

## 一句话总结

Chat Completions 是"对话式 API"——你说话，AI 回话，简单直接。Responses API 是"任务式 API"——你提需求，AI 全程搞定，省心但控制感弱一些。两者不是替代关系，而是**分工关系**：简单聊天用旧的，复杂任务用新的。就像你不会用同一个工具修水管和写代码——选对工具比追新更重要。

---

> 两代 API 的并存不是混乱，而是需求分层的结果。对话场景和任务场景本来就是两类不同的问题，需要两套不同的接口设计。理解了"对话 vs 任务"的本质区别，选型就是直觉——你的应用是聊天窗口还是项目管理器？答案决定了你该用哪个 API。
